# Product Data V2 — Visual Readiness Report

> **Scope.** Assesses whether KOLEEX can currently support each visual product-knowledge surface defined in [Visual Product Experience](./architecture/visual-product-experience.md). The verdict is about the **V2 product-knowledge pipeline** (schema + dictionaries + projections + renderers), not the legacy app. Where the live app already has a relevant building block, it is credited under "What exists today" — but a legacy renderer fed by legacy data is **not** the same as the V2 surface being ready.
>
> **Status of inputs.** Per `architecture-freeze-v1.0.md` and `product-data-v2-schema.md`, the V2 schema is **APPROVED IN PRINCIPLE but NOT APPLIED** — no `pd_` tables, no `pd_facets`, no `pd_search_doc`/`pd_ai_doc` projections exist in production. Confirmed by inspection: a grep over `supabase/migrations/` for `pd_facets | pd_skus | pd_product_types | pd_search_doc | pd_ai_doc | pd_compatibility | pd_families` returns **nothing**. Stage 2+ is gated on (a) the validated production baseline (Stage 1.5) and (b) prefix-freeze sign-off.

---

## The three-layer test

Every verdict separates three distinct things, because "we have a design" is routinely mistaken for "it works":

1. **Design exists** — the visual metadata vocabulary and per-entity cards are specified in `visual-product-experience.md §17`. This is **complete** for all eight surfaces. Design is *not* a gating factor anywhere; it is the one thing fully in hand.
2. **Data populated** — the metadata fields (`icon_key`, `image_role`, `presentation_group`, `spec_card_priority`, `comparison_display`, `*_display_hint`, `ai_visual_hint`) actually exist as columns AND carry values. This is **missing everywhere** — the columns are design-only field names, not applied DDL, and even the dictionaries that would supply the values (e.g. per-facet presentation hints) are not built.
3. **Renderer exists** — a UI that reads the metadata/projection and draws the surface without per-product hardcoding. This is **partially true only for spec cards / product pages** via `ProductPreview.tsx`, and even there it reads the *legacy* `schema_specs`/PTE store, not the V2 `pd_facets` projection.

A surface is **READY** only when all three are true. Today none clear that bar, because Layer 2 is empty across the board.

## Common blockers (shared prerequisites)

These are the cross-cutting gates. They are ordered by how many surfaces they unblock.

1. **V2 schema is design-only / not applied.** No `pd_` tables exist in production (confirmed: zero `pd_*` migrations). The Visual Presentation Metadata layer (§17) is a list of *design-level field names*, not columns — `icon_key`, `image_role` (full vocabulary), `hero_image`/`gallery_images`/`diagram_image`, `spec_card_priority`, `presentation_group`, `comparison_display`, `ai_visual_hint`, `quotation_display_hint`, `website_display_hint`. Until the schema is applied, **no surface can reach Layer 2 (data populated)**. This single blocker holds back all eight.

2. **No read-model projections (`pd_search_doc`, `pd_ai_doc`).** Every customer-facing surface is designed to render from a cached, lookup-free projection (flattened facets, resolved media, embedded visual refs). Cards, comparison, quotation lines, website pages, and especially AI answers all depend on it. Without the projections, even populated facets cannot be rendered "without a developer hardcoding it" — which violates the §1 visual-first principle.

3. **Facet/Type/Device dictionaries lack presentation hints.** `facet-dictionary-master.md`, `device-dictionary-master.md`, and `application-dictionary-master.md` define the vocabulary but carry **no** `presentation_group`, `spec_card_priority`, display style, `comparison_display`, or `icon_key` values yet. The governance bar in `visual-product-experience.md §19` ("visuals are part of done") is a *future* gate, not a satisfied one. Spec Cards, Comparison Cards, and Compatibility Cards cannot become self-describing until these hints are authored.

4. **Partial photo + icon plumbing, not the full resolution spine.** The live app has `product_media.role` (a **subset** of `image_role`) and a classification-level icon registry (`visual_types`/`visual_assets`), but **not** entity-level `icon_key` (Model/SKU/Facet/Device/Part) and **not** the §3 upward resolution order. So "nothing renders blank" is not yet guaranteed for V2 entities.

5. **Upstream stage gates.** Per the freeze, Stage 2+ (where any of this lands) is blocked on **(a)** a validated production baseline on a clean branch (Stage 1.5) and **(b)** prefix-freeze sign-off. No schema, projection, or dictionary-hint work can be applied to prod ahead of these.

### What flips the most surfaces to READY

- **Apply the `pd_` schema + the projections (blockers 1 + 2)** unblocks Layer 2 for **all eight** surfaces simultaneously — it is the master gate.
- **Author the facet presentation hints (blocker 3)** is the highest-leverage *content* step: it directly flips **Spec Cards, Comparison Cards, and Website Product Pages** toward READY, because the renderers either exist or are a small step from existing.

### Closest to READY today

**Spec Cards** and **Website Product Pages** are tied for closest. Both already have a working, schema-driven, presentation-only renderer (`ProductPreview.tsx` on `/products/preview/[slug]`) that proves the §8/§13 pattern end-to-end on legacy data. They need only **(1)** the `pd_facets` schema + projection applied and **(2)** facet presentation hints populated, then a rebind of the existing renderer — no net-new UI. Every other surface needs both data AND a renderer that does not yet exist.
